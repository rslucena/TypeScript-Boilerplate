Architecture Decisions ‚Äã

This document records the architectural decisions made for the project, providing context on "why" certain technologies or patterns were chosen.

ADR-001: Runtime Environment ‚Äã

  • Status: Accepted
  • Decision: Use Bun instead of Node.js.
  • Context: We needed a high-performance runtime that simplifies the toolchain.
  • Consequences:
    • ‚úÖ Pros: Faster startup, built-in test runner, built-in package manager (no need for npm/yarn), TypeScript support out of the box.
    • ‚ö†Ô∏è Cons: Some Node.js APIs might have partial compatibility (though improving daily).

ADR-002: ORM (Object-Relational Mapping) ‚Äã

  • Status: Accepted
  • Decision: Use Drizzle ORM.
  • Context: Validation of TypeORM (legacy/heavy) vs Prisma (slow cold start) vs Drizzle.
  • Consequences:
    • ‚úÖ Pros: Lightweight, "If you know SQL, you know Drizzle", excellent type safety, no runtime overhead.
    • ‚ö†Ô∏è Cons: Manual migration management is slightly more verbose than Prisma's auto-migrate-dev.

ADR-003: Web Framework ‚Äã

  • Status: Accepted
  • Decision: Use Fastify.
  • Context: Express is unmaintained/slow. NestJS is too opinionated/heavy for this micro-service goal.
  • Consequences:
    • ‚úÖ Pros: Highest performance in Node ecosystem, schema-based validation fits perfectly with Zod, great plugin ecosystem.
    • ‚ö†Ô∏è Cons: Learning curve for hooks/plugins if coming from Express.

ADR-005: Granular Error Handling ‚Äã

  • Status: Accepted
  • Decision: Implement specific HTTP status codes (e.g., 409 Conflict) instead of generic ones for resource state conflicts.
  • Context: Relying on generic errors like 422 for everything makes it harder for API clients to handle specific business logic exceptions properly.
  • Consequences:
    • ‚úÖ Pros: Better client-side error handling, clearer API documentation, follows REST best practices.
    • ‚ö†Ô∏è Cons: Slightly more verbose action logic to handle specific conflict scenarios.
